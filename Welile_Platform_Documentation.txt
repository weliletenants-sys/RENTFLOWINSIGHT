1. Executive Summary
Welile Technologies Limited is a rent-guarantee and rent-facilitation platform designed to scale across Africa and globally. The platform acts as a system operator and guarantor, facilitating rent payments between tenants, landlords, supporters (investors), and field agents.
The platform earns revenue through access fees, registration fees, and financial intermediation — it does not sell loans. Revenue is recognized incrementally as collections occur (Recognized vs. Deferred), with no upfront or estimated recognition.
Core Capabilities:
• Rent guarantee and disbursement to landlords
• Investment portfolio management (Supporter ROI, Angel Pool)
• Field agent network with float management and commissions
• Ledger-based double-entry financial system
• Progressive Web App (PWA) with offline-first design
• 15 distinct user roles with role-based access control

6.2 Role Architecture
The platform has 15 distinct roles stored in a dedicated user_roles table (never on profiles). Roles are divided into public and staff categories:
CategoryRolesAccess MethodPublic (self-assignable)tenant, agent, landlord, supporterUsers can freely switch between theseStaff (backend-assigned)manager, employee, operationsRequires admin assignmentExecutiveceo, coo, cfo, cto, cmo, crmRequires admin assignment + access codeSystemsuper_admin, hrHighest privilege, backend-only assignment6.3 Role Switching (Atomic)
The auth system uses a ref-synchronized approach to prevent race conditions during role switching:
1. User taps a public role in the bottom role switcher
2. A pendingRole state is set immediately (UI shows loading)
3. grantAndSwitchRole() atomically: (a) adds role if missing, (b) updates rolesRef, (c) sets active role
4. Dashboard renders only when role matches pendingRole
Staff/admin roles are NOT shown in the switcher unless the user already possesses them.
6.4 RoleGuard Component
Protected routes use the RoleGuard component which: (1) checks if any of the user's roles match the allowedRoles, (2) redirects unauthenticated users to /auth, (3) redirects unauthorized users to /dashboard, and (4) logs unauthorized access attempts to audit_logs.


7. Financial Architecture
The financial system implements a 4-layer security fortress ensuring absolute integrity of all money movement.
7.1 The Golden Rule
Cash = Wallet + Platform. Every financial operation must maintain this invariant through double-entry accounting.
7.2 The Mandatory Pipeline
CLIENT ? Edge Function (validation) ? create_ledger_transaction RPC ? general_ledger ? sync_wallet_from_ledger trigger ? wallets.balance
Direct wallet mutations are FORBIDDEN. The trg_guard_wallet_mutation trigger blocks all balance changes unless the wallet.sync_authorized session flag is set to true — a flag only granted to the sanctioned sync_wallet_from_ledger trigger.
7.3 create_ledger_transaction RPC
The central gatekeeper for all financial writes enforces:
• Atomic transactions: every action uses a unique transaction_group_id where SUM(cash_in) == SUM(cash_out)
• Deterministic idempotency: uses pg_advisory_xact_lock acquired BEFORE lookups
• Balance guard: blocks wallet-scope cash_out if ledger-derived balance is insufficient
• Category allowlist: only locked categories accepted in strict mode
• Audit trail: every transaction group logged with metadata for CFO traceability

• Minimum liquidity threshold (15%) — blocks withdrawals if breached
• Strict mode — rejects any category not in the locked allowlist
• Automated reconciliation checks
• Real-time drift detection between wallets and ledger


8. Key Business Flows
8.1 Rent Approval Pipeline
A rent request goes through a multi-stage approval pipeline:
5. Tenant submits rent request (or agent submits on behalf)
6. Manager reviews and validates (landlord verification, property check)
7. Financial operations approves (checks platform solvency)
8. Disbursement initiated (creates ledger entries, updates rent_receivable)
9. Agent delivers cash to landlord (GPS-verified, photo evidence)
10. Delivery confirmed (landlord signature/confirmation)
8.2 Tenant Repayment
• Agent collects from tenant (cash or mobile money)
• Collection recorded with payment method and evidence
• Ledger entries created: tenant_repayment (cash_in) + rent_principal_collected
• Agent commission earned (if applicable) with 72-hour lock
• Wallet balances updated via trigger
8.3 Supporter ROI Flow
• Supporter deposits into portfolio
• ROI calculated based on portfolio performance
• ROI payments processed via process-supporter-roi edge function
• Ledger entries: roi_expense (platform) + roi_wallet_credit (supporter)
• Blocked if platform liquidity < 15%
8.4 Angel Pool Investment
• Shares priced at UGX 20,000 each, max 25,000 shares total
• Total pool equity: 8% of company
• Ownership: Pool % = (shares / 25,000) * 100; Company % = (shares / 25,000) * 8
• Agents earn 1% platform-funded commission on facilitated investments
• Investment via angel-pool-invest or agent-angel-pool-invest edge functions
8.5 Agent Commission Rules
• Commissions earned as wallet credits (platform cash_out)
• 72-hour lock period before withdrawal
• Can be used to cover tenant defaults AFTER: (a) 72h grace period AND (b) agent float exhausted
• Categories: agent_commission_earned, agent_commission_withdrawal, agent_commission_used_for_rent


9. Security Architecture
9.1 Row-Level Security (RLS)
ALL 145+ tables have RLS enabled. Policies enforce:
• Users can only read/write their own data (user_id = auth.uid())
• Staff access controlled via has_role() security-definer function
• No anonymous access to sensitive tables (financials, portfolios)
• Scoped read policies (e.g., location_requests only visible to involved parties)
9.2 Financial Security Layers
LayerMechanismProtection1. Edge FunctionAuth + role validationBlocks unauthenticated/unauthorized requests2. RPC Gatekeepercreate_ledger_transactionEnforces double-entry, category allowlist, balance checks3. Trigger Guardtrg_guard_wallet_mutationBlocks direct wallet balance edits4. Sync Triggersync_wallet_from_ledgerOnly authorized path to update wallet balances9.3 Audit & Compliance
• audit_logs table records all admin actions with user ID and metadata
• system_events table logs all state transitions
• Financial events are permanent; operational logs follow 7-day retention
• Unauthorized access attempts logged with path and attempted roles
9.4 Anti-Patterns (Forbidden)
• Direct wallet edits (.update({ balance }))
• Business logic in UI components
• Offline financial updates
• Manual balance adjustments or fix balance buttons
• Duplicate logic across edge functions
• Anonymous access to financial data
