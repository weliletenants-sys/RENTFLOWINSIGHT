---
name: Wallet sole-writer rule
description: apply_wallet_movement is the ONLY function that mutates wallet bucket fields. sync_wallet_from_ledger is intentionally a no-op since 2026-04-23 to stop double-counting.
type: constraint
---

**Rule.** Only `apply_wallet_movement(user_id, category, amount, direction)` may write to `wallets.balance / withdrawable_balance / float_balance / advance_balance`. It is invoked by `create_ledger_transaction` for every ledger entry.

**Why.** Two triggers used to update `wallets.balance` independently per ledger insert:
- `trg_sync_wallet_from_ledger` → `sync_wallet_from_ledger()` (legacy raw delta)
- `apply_wallet_movement()` (bucket-aware: balance = withdrawable + float)

This double-counted every movement and was the root cause of the recurring phantom drift the 15-min cron has been logging.

**Fix in place since 2026-04-23 (3 migrations).**

1. `sync_wallet_from_ledger()` body is now `RETURN NEW;` only. The trigger and function symbol are kept attached for backward compatibility — DO NOT delete or "restore" them without first auditing every dependent migration.
2. `enforce_non_negative_balance()` still floors negative balances to 0, but now logs every clamp into `public.wallet_overdraw_events` (RLS: super_admin/cfo/coo/operations read). Previously-silent overdraws are now attributable.
3. `apply_wallet_movement()` no longer silently returns when a category has `route='none'`. It logs the unrouted attempt into `public.wallet_unrouted_movements` (same RLS) so structural routing gaps surface immediately.

**Diagnostic tables added.**
- `wallet_overdraw_events` — every time the floor clamped a negative balance.
- `wallet_unrouted_movements` — every category that hit `apply_wallet_movement` with no bucket route. The two known offenders to migrate next: `test_funds_cleanup`, `proxy_investment_commission`.

**Do not.**
- Do not re-enable any logic inside `sync_wallet_from_ledger()`.
- Do not raise from `apply_wallet_movement` when `route='none'` — log only, never block live edge functions.
- Do not bypass `apply_wallet_movement` to set wallet fields directly. The wallet write-lockdown trigger will reject it.
